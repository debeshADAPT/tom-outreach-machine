## Prioritized list for a future fix-task (not implemented in this pass)

1. **Highest priority — user-trust risk.** Add explicit in-UI "Preview / not yet sent" or "Mock data" indicators to `EmailLogsTab.tsx`, the match-score badge in `ProspectsTable.tsx` and `ProspectDrawer.tsx` (plus its mock replies), and all of `AIInsightsTab.tsx`. These are the only findings where a real user could be actively misled into believing something has already happened (email delivery, AI-driven prioritization, an actual prospect reply) when it hasn't.
2. **Data-integrity fix.** Make `insertAiContextProspects` dedup race-safe — either confirm migration 009's unique index is actually applied in production and add proper unique-violation handling, or switch to an `upsert`/`onConflict` pattern so correctness no longer depends on a manually-run SQL file.
3. **Reliability fix.** Add a lightweight staleness check for `brief_status`/`intelligence_status` stuck at `'pending'` — e.g. compare an "entered pending at" timestamp against a threshold and surface a "may have stalled — resync" affordance, instead of relying on an admin noticing and manually resyncing.
4. **Documentation correction.** Update `ai/HANDOVER.md` to remove/correct the "Generate Context spinner — partial fix in progress" note; current code shows this already resolved.
5. **Low-risk cleanup.** Delete `lib/supabase.js`; factor the five-times-duplicated `auth.users`-via-`profiles` lookup pattern into a shared `lib/profiles.ts` helper.
6. **Low priority.** Wire up or remove the dead "Follow-up" button in `AIInsightsTab.tsx`; decide whether `startProspectSequence`/`bulkStartSequences`/`saveSequenceDelays` should be deleted outright or intentionally kept dormant pending a real "start sequence" UI (if kept, recommend adding a code comment warning about live-send implications before anyone wires a button to them).
